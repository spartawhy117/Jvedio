# 测试目标文档

## 1. 背景

当前项目已经从“零散手工测试 + 少量 UI 测试”转向“数据驱动 + 非 UI 自动化测试”主线。

测试体系当前重点围绕以下主线能力建立：

- MetaTube 唯一搜刮源
- 影片详情转换
- 演员头像获取
- sidecar 文件输出
- 正式缓存写入
- 扫描库自动整理平铺影片
- 启动日志覆盖策略

本文件用于定义：**测什么、为什么测、通过标准是什么。**

## 2. 测试目标总览

当前测试目标分为三大类：

- 快速验证
- 网络验证
- 扫描链验证

每一类对应不同的验证目标和断言强度。

## 3. 快速验证目标

快速验证的目标是：
- 在不依赖网络的前提下，验证核心路径和规则是否回归

主要覆盖：

- sidecar 路径解析
- actor-avatar 路径解析
- MetaTube cache 读写
- `ScrapeResult -> Dictionary` 映射
- 自动整理目录的命名回退逻辑
- 日志覆盖逻辑

### 成功标准
- 路径命名正确
- 缓存可读写
- 映射字段完整
- 目录整理规则稳定
- 启动日志覆盖行为符合预期

## 4. MetaTube 网络验证目标

MetaTube 网络验证的目标是：
- 确认远程服务可达
- 确认影片搜索、详情、演员搜索、演员详情链路工作
- 确认测试输出目录生成结果完整
- 确认日志信息足够定位问题

主要覆盖：

- warmup
- `/v1/movies/search`
- `/v1/movies/{provider}/{id}`
- `/v1/actors/search`
- `/v1/actors/{provider}/{id}`
- 测试输出写入

### 成功标准
- 根地址与 `/v1/providers` 可达
- 输入 `VID` 后能命中影片
- 详情能转换为 `ScrapeResult`
- 至少一个演员头像成功（弱断言）
- 测试输出目录中生成：
  - `meta.json`
  - `*.nfo`
  - 海报/缩略图/背景图文件

## 5. 扫描链验证目标

扫描链验证的目标是：
- 验证平铺影片在扫描前被自动整理到独立目录
- 验证字幕跟随迁移
- 验证整理失败时跳过影片且不继续搜刮
- 验证整理成功后新路径进入后续链路

主要覆盖：

- 平铺视频整理
- 字幕迁移
- 失败跳过
- `Video.Path` 更新
- `ScanResult.NotImport` 记录

### 成功标准
- 平铺影片能创建独立目录
- 同名字幕一起移动
- 整理失败的影片进入跳过结果
- 整理成功后路径已变更

## 6. sidecar 与缓存目标

当前正式运行的目标规则：

### 正式影片目录
- `<VID>.nfo`
- `<VID>-poster.jpg`
- `<VID>-thumb.jpg`
- `<VID>-fanart.jpg`

### 正式缓存目录
- `data/<user>/cache/video/<VID>.json`
- `data/<user>/cache/actor-avatar/<actorId>.jpg`

### 测试输出目录
- `data/<user>/log/test/<VID>/`

### 成功标准
- 正式 sidecar 命名统一
- 正式缓存路径统一
- 测试输出不污染正式缓存

## 7. 日志目标

当前日志目标：

- 每次启动覆盖当日日志
- MetaTube 测试日志并入主日志流
- 扫描整理日志具备语义前缀
- 测试输出和主日志都可用于定位问题

### 成功标准
- 主日志文件每次启动后清空旧内容
- MetaTube 测试链日志清晰
- `[Library-Organize]` 日志可定位整理失败原因

## 8. 强断言与弱断言

### 强断言
这些测试必须稳定通过：

- 根地址可达
- providers 可达
- 电影搜索有结果
- 详情可返回
- sidecar 输出目录可写
- 自动整理成功时目录结构正确
- 整理失败时影片被跳过

### 弱断言
这些测试不建议写成“必须全成功”：

- 每个演员都必须有头像
- 每张图片都必须成功下载
- 每个 provider 的字段完全一致

### 当前建议
演员头像采用：
- **至少一个演员头像成功**
作为默认弱断言

## 9. 当前未完全覆盖的目标

虽然当前 18 个测试已经通过，但以下目标仍可继续增强：

- 多分段视频整理
- actor detail 与别名匹配增强
- 图片源 403/超时的容错验证
- 正式扫描链与 MetaTube 主链联动验证
- 正式 sidecar 输出后的读取一致性验证

## 10. 文档更新规则

当测试目标发生变化时，必须更新本文件。

需要更新的情况包括：

- 新功能带来新的测试目标
- 老功能新增新的验收边界
- 强断言 / 弱断言策略变化
- 覆盖范围发生变化
