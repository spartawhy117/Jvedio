### F-038 点击播放影片后已成功调用系统播放器，但前端仍提示播放失败

现象：
- 点击播放影片后，系统默认播放器已经被成功拉起，影片也开始正常播放。
- 但前端随后仍弹出“播放失败”提示，当前反馈为 `HTTP 500: Internal Server Error`。
- 这说明“实际播放动作”与“接口成功回执”已经发生了状态分裂。

已核对证据：
- `dotnet/Jvedio.Worker/Services/VideoService.cs` 中 `PlayVideo(...)` 当前在成功启动播放器后，仍继续构造包含 `LaunchedAtUtc`、`LastPlayedAt`、`UsedPlayerPath`、`UsedSystemDefault`、`VideoId` 的返回对象，并记录 `request.PlayerProfile`、`request.Resume`。
- `tauri/src/api/types.ts` 中前端当前声明的 `PlayVideoResponse` 只包含 `played` 和 `playerUsed` 两个字段，`PlayVideoRequest` 也只保留 `playerPath`。
- 先前后端 API 校验基线里，`/api/videos/{id}/play` 已经被当作 `Expected status 500` 处理，说明这个端点的异常并不是新近偶发，而是当前链路里已存在的已知回归。

根因判断：
- 初步看不是“系统播放器没有成功打开”，而是“播放器已经打开后，Worker 在返回成功响应前又触发了异常”。
- 最高概率是播放接口仍残留旧版请求 / 响应契约字段，导致成功启动后在日志写入、响应对象构造或序列化阶段抛错，最终把一次成功播放误报成 HTTP 500。
- 因此前端 toast 更像是被后端错误回执误导，而不是前端自身把成功结果错误翻译成失败。

建议解决方案：
- 先统一 `PlayVideoRequest` / `PlayVideoResponse` 的前后端契约，清理 `PlayerProfile`、`Resume`、`LaunchedAtUtc` 等旧字段残留。
- 收口 Worker 播放成功路径：只要 `Process.Start(...)` 或系统默认打开已成功，就必须返回稳定的成功 envelope，不能在后续附加字段上再让接口失败。
- 为 `/api/videos/{id}/play` 增加或更新自动化测试，覆盖“系统默认播放器成功拉起时返回 200 + played=true”的场景，移除对 500 的旧基线依赖。

落地顺序：
- 先检查 `VideosController`、`VideoService` 与 contracts 中的播放 DTO 是否一致。
- 再修正前后端播放响应模型与日志字段。
- 最后补自动化测试并回归详情页、列表页和喜欢页的播放入口提示。

状态：
- `待修复`

### F-039 影片详情页在大屏下空间利用不足，sidecar 标签层级与 fanart 展示方式都不对

现象：
- 当前影片详情页在拉到 4K 分辨率后，右侧内容区仍然大量留白，整体自适应范围明显偏小。
- 四个数据抓取标签 `NFO / 海报 / 缩略图 / 同人图(fanart)` 当前仍在左侧海报与 `VID` 下方，反馈希望改成横向排布到名称下面。
- 当前不应继续保留左侧前景图，详情页顶部应直接使用 `xxxx-fanart.jpg` 作为“演员标题以上所有内容”的底部背景板。

已核对证据：
- `tauri/src/pages/pages.css` 中 `.video-detail-layout` 当前仍是 `220px + 1fr` 的两列结构，左侧海报列宽固定，天然更偏向中等屏宽而不是 4K 横向展开。
- `tauri/src/pages/VideoDetailPage.tsx` 当前 sidecar 状态块仍在左列 `VID` 下方，以 `.sidecar-status-grid` 单独渲染，尚未进入标题区。
- `tauri/src/pages/pages.css` 中 `.sidecar-status-grid` 当前仍是 `repeat(2, minmax(0, 1fr))` 的 2x2 网格，不是名称下方的一行横向标签。
- `tauri/src/pages/VideoDetailPage.tsx` 当前详情图来源仍是 `thumbUrl ?? posterUrl`，并未建立 fanart 专用背景层；也就是说当前结构里既没有“顶部信息区底部背景板”这一层，也仍在保留左侧前景图语义。
- `tauri/src/App.css` 与详情页样式当前仍保留共享内容区限宽思路，这会进一步放大大屏留白问题。

根因判断：
- 当前详情页仍是“固定左侧海报列 + 右侧正文列”的中屏结构，不适合 4K，导致大量留白。
- sidecar 标签当前仍被当成左侧海报附属信息，所以还在 `VID` 下方，而不是标题信息区。
- 这不是单个 margin 或 width 数值过小，而是详情页仍沿用“左侧独立海报列 + 右侧正文列”的中屏信息页模型，没有切换到适合宽屏的“顶部内容层 + 背景图层”结构。
- sidecar 标签之所以放错位置，是因为它们当前仍被视为左侧海报附属状态，而不是标题信息的一部分。
- fanart 没有成为背景板，不是资源缺失，而是页面当前仍默认保留左侧前景图语义，只实现了 `thumb/poster` 的前景展示，没有切换到“取消左侧前景图、改为 fanart 顶部背景层”的结构。

建议解决方案：
- 重构详情页顶部结构：取消左侧前景图，标题、元数据、主操作、sidecar 标签组成前景内容层，fanart 作为该区域底层背景板，覆盖到演员标题上方为止。
- 将 sidecar 标签从左侧信息列移到标题区下方，改为一行可换行的横向 badge 组，按 `NFO / 海报 / 缩略图 / fanart` 顺序展示。
- 图片职责重新拆分：列表卡仍保持 `poster`，详情页不再保留独立左侧主图，顶部背景层优先走 `fanart`，缺失时再决定是否降级到其他图。
- 大屏布局从“固定左列”改成“更宽的主内容栅格 + 顶部背景区域 + 下部内容区”，移除不必要的全局限宽，让 2K/4K 下元数据区、按钮区和路径区能真正展开。

落地顺序：
- 先梳理 `VideoDetailPage.tsx` 的 DOM 结构，把标题区、sidecar 标签区、fanart 背景层拆开。
- 再重写 `pages.css` 中详情页顶部布局与大屏断点规则，优先解决 1440p / 4K 留白问题。
- 最后回归 fanart 存在 / 缺失、单演员 / 多演员、长标题、窄窗与超宽屏几类详情页样本。

状态：

### F-040 高分辨率下设置页面底部动作按钮没有锚定在右下角

现象：
- 在高分辨率或超宽窗口下，设置页底部的“恢复默认 / 保存设置”按钮虽然位于表单底部，但没有稳定锚定在活动页右下角。
- 当前表现是按钮组跟随内容列宽停留在中间偏左区域，右侧仍保留明显空白，与“活动页底部固定动作区”的预期不一致。

已核对证据：
- `tauri/src/pages/SettingsPage.tsx` 当前仍在 `.settings-form-inner` 末尾直接渲染 `<div className="settings-actions">`，没有挂到共享的 `page-activity-footer` 一类底部固定槽位。
- `tauri/src/pages/pages.css` 中 `.settings-form-inner` 当前宽度被限制为 `width: min(100%, 760px)`，这会让按钮组即便 `justify-content: flex-end`，也只能在 760px 的内容列内部右对齐，而不是对整个活动页右下角对齐。
- `.settings-actions` 当前只是 `margin-top: auto` 的普通流内尾部区域，并没有 `sticky` footer 或独立 shell footer 的锚定机制。

根因判断：
- 根因不是按钮自身样式，而是设置页依旧采用“限宽表单列 + 流内尾部按钮”的旧结构。
- 在高分辨率下，内容列被限制在 760px 宽，动作区自然只能跟着这条窄列走，无法锚定到右侧活动页的真实右下角。
- 这说明设置页还没有真正接入统一的“活动页正文区 + 底部固定动作区”布局模型。

建议解决方案：
- 将设置页动作按钮从 `.settings-form-inner` 的普通流末尾迁移到共享的底部 footer 区，和分页组件同样使用稳定的底部锚点思路。
- 将“正文阅读限宽”和“页面底部动作区占满可用宽度”拆开：表单内容可继续限宽，但底部动作条必须基于整个活动页宽度右对齐。
- 如设置页继续保留独立实现，也至少应引入 `settings-footer-shell` 之类的全宽底部容器，保证在 2K / 4K 下按钮始终贴近右下角。

### F-041 CPU 占用率与内存使用量统计不准确，前端和 Worker 的总用量没有正确合并

现象：
- 当前侧栏“运行状态”中的 CPU 占用率和 Memory 数值与用户体感不一致，反馈认为统计不准确。
- 当前需求是这两个指标应展示“前端 + Worker”合计后的总占用，而不是只看单一进程。

已核对证据：
- `tauri/src-tauri/src/worker.rs` 当前 `RuntimeMetricsPayload` 已包含 `shellCpuPercent`、`workerCpuPercent`、`totalCpuPercent`、`shellMemoryMb`、`workerMemoryMb`、`totalMemoryMb`。
- 同文件 `get_runtime_metrics()` 当前的总量计算方式是 `shell + worker`，也就是只把 Tauri shell 主进程与 `Jvedio.Worker` 子进程相加。
- `tauri/src/App.tsx` 当前侧栏展示已经直接绑定 `metrics?.totalCpuPercent` 和 `metrics?.totalMemoryMb`，说明前端 UI 并不是只显示 Worker 数值，而是在显示当前后端返回的“shell + worker”结果。
- 当前 Tauri 侧使用的是 `sysinfo 0.30.13`；其 `Process::cpu_usage()` 官方定义是“进程总 CPU 使用率”，在多核机器上该值本来就可能大于 `100%`，如果要得到 `0%-100%` 的整机口径值，需要再除以 CPU 数量。

根因判断：
- 当前问题不应简单理解为“拿到了单线程值”，更准确的说法是：现在拿到的是 `sysinfo` 的进程总 CPU 使用率原始值，但还没有按整机 CPU 线程 / 逻辑核心数做归一化。
- 如果当前产品要展示的是用户直觉里的“整机 CPU 占用率（0%-100%）”，那么直接把 `shell_cpu_percent + worker_cpu_percent` 原样显示会放大数值口径，尤其在多核机器上会与任务管理器常见口径不一致。
- 此外，问题仍可能叠加另一层误差：当前 total 只统计了 Tauri shell 主进程与 Worker，未必覆盖 WebView 渲染进程，因此“口径未归一化”和“前端进程可能漏统”都需要一起核对。

建议解决方案：
- 先明确 CPU 指标口径：如果 UI 要对齐整机 0%-100% 口径，应先把聚合后的原始 `cpu_usage` 除以逻辑 CPU 数。
- 在完成归一化后，再核对“前端整体”是否只统计到了 shell 主进程；若 WebView 渲染进程未计入，需要继续补充前端进程聚合。
- 内存指标不需要做 CPU 核心数归一化，但仍要确认是否遗漏前端渲染进程的实际内存占用。

落地顺序：
- 先补充 CPU 口径确认：读取逻辑 CPU 数，并验证 `sysinfo` 原始值在当前机器上的换算关系。
- 再确认当前运行时前端实际对应的进程集合，验证 WebView 渲染进程是否未被统计。
- 最后扩展 `get_runtime_metrics()` 的采样与换算逻辑，回归空闲态、扫描态、详情页浏览态等不同负载下的 CPU / 内存表现。

状态：
- `待修复`

### F-042 演员头像缓存命名不可读，且缓存应保留并按演员名规范化复用

现象：
- 当前 `cache/actor-avatar/` 里的演员头像文件名并不是按程序中显示的演员名字保存，而是出现 URL 编码串或其他不可读 key，目录可读性很差。
- 当前删除某演员的全部信息后，`cache/actor-avatar/` 下对应头像文件不会被删除。
- 当前策略决定为“缓存保留不删除”，但下次重新索引到相同演员时，应确保直接复用本地缓存，不再重复拉取。

已核对证据：
- `dotnet/Jvedio.Worker/Services/LibraryScrapeService.cs` 中 `BuildActorAvatarCacheKey(...)` 当前优先取 `avatarUrl` 提取的 key，其次取 `actorId`，最后才回落到 `actorName`，这会直接产生 URL 派生 key 或 ID 风格文件名。
- `dotnet/Jvedio.Worker/Services/ActorService.cs` 中 `GetActorAvatarCacheKeys(...)` 当前查询顺序同样包含 `actorId`、从 `imageUrl/webUrl` 提取的 key 和名字 fallback，说明运行期查找本身就依赖多种历史 key。
- 当前已见到 `ResolveActorAvatarPath(...)` / `WriteActorAvatarAsync(...)` 的缓存读写逻辑，但未见到“删除演员后清理 `cache/actor-avatar`”的显式链路，说明现阶段缓存本身就是偏保留型设计，只是命名与复用规则没有收口好。

根因判断：
- 这是“缓存 key 设计偏向抓取源稳定性”与“用户可读性 / 长期复用策略”之间没有收口造成的问题。
- 当前命名策略优先考虑 URL 或 actorId，便于避免同名演员冲突，但直接牺牲了缓存目录可读性，也让用户看到一堆编码文件名。
- 当前既然决定保留缓存，那么问题重点就不再是“删后是否清理”，而是“删后 / 重建索引后能否稳定复用同一演员的旧缓存”。
- 如果复用命中仍依赖 URL key 或 actorId 漂移，就会出现缓存留着但又重复下载的问题。

建议解决方案：
- 采用固定策略：演员头像缓存不删除，作为可复用资源长期保留。
- 缓存文件命名改为“演员展示名规范化 + 必要时稳定去重后缀”，避免继续使用 URL 编码串或纯 actorId 作为主文件名。
- 重新索引相同演员时，必须先按规范化演员名策略命中本地缓存；只有本地确实不存在可用头像时才允许重新下载。
- 如果担心同名演员冲突，可采用“规范化演员名 + 哈希短后缀”的可读命名，而不是完全依赖 URL key。

落地顺序：
- 先梳理演员头像缓存的写入 key、读取 key 和删除链路，确定当前哪些入口会生成或遗留头像文件。
- 再将缓存生命周期固定为“长期保留并强制复用”，删除演员记录时不删除头像缓存。
- 最后统一缓存命名规则，并回归“首次抓取 / 删除演员后重建索引 / 重新索引同名演员 / 同名不同演员”几类场景。

状态：
- `待修复`
