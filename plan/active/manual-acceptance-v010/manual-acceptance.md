## 当前活跃问题

### F-028 右侧活动页整体不是自适应大小

现象：
- 当前所有右侧活动页整体不是自适应大小。
- 以设置页为例，右侧活动区被限制在较窄内容宽度里，窗口放大后仍保留大块空白。

已核对证据：
- `tauri/src/App.css` 中 `.page-content-section` 当前写死了 `max-width: 900px`。
- `tauri/src/pages/pages.css` 中 `.settings-form-area` 当前又额外写死了 `max-width: 640px`。
- `tauri/src/App.css` 中 `.content-area` 只是整块滚动容器，没有为活动页提供“内容区 + 底部固定区”的 shell 分层。

根因判断：
- 根因不是单页样式失误，而是活动页共享容器同时承担了“页面整体占满可用空间”和“正文阅读限宽”两种职责。
- 结果是外层壳层先被 `900px` 限宽，设置页内层又被 `640px` 二次限宽，右侧活动区自然无法随窗口放大。

建议解决方案：
- 将活动页结构拆成两层：
- 外层 `page-activity-shell` 负责 `flex: 1`、`min-height: 100%`、`min-width: 0`，始终铺满右侧可用区域。
- 内层 `page-readable-content` 只在需要阅读限宽的页面中使用，负责正文最大宽度。
- `page-content-section` 不再承担全局 `max-width`，只保留 padding。
- 设置页改成“左导航 + 右表单区”全宽布局，表单正文另包一层 `settings-form-inner` 做可选限宽。

落地顺序：
- 先改 `App.css` 和 `pages.css` 的共享容器定义。
- 再把设置页、库页、喜欢页、演员页统一迁移到新的活动页 shell。
- 最后配合 `F-033` 把分页区挂到 shell 底部固定区域。

状态：
- `已实现，待人工复核`

### F-031 同步 100% 后进度条与按钮状态没有及时切换

现象：
- 当前库项已经显示同步 `100%`，并且完成明细已经达到 `2/2`。
- 但完成度进度条仍停留在 `0%`，操作区按钮也仍显示“同步中 100%”。

已核对证据：
- `tauri/src/pages/LibraryManagementPage.tsx` 中按钮、badge 和文案都优先读 `activeTask`，其次才读 `lib.isFullySynced / lib.completionPercent`。
- 同文件里 `useOnLibraryChanged(() => { /* no additional action needed */ })` 当前没有触发 `tasksQuery.refetch()`。
- `activeTasksByLibrary` 完全来自 `tasks:list` 查询，而库 DTO 的 `completionPercent / isFullySynced` 来自 `BootstrapContext` 中的库列表，两者刷新链路并不一致。
- `dotnet/Jvedio.Worker/Services/LibraryService.cs` 已会返回 `SyncedVideoCount / CompletionPercent / IsFullySynced`，说明后端库状态本身已具备完成态数据。

根因判断：
- 根因不是后端没有算出 100%，而是前端把“任务列表状态”和“库完成态状态”拆成了两套来源。
- 当 `library.changed` 已刷新库 DTO，但 `tasks:list` 仍停留在旧的 running 任务时，界面就会出现“文本 100%，按钮仍同步中”的状态撕裂。

建议解决方案：
- 在库管理页监听 `library.changed` 时同步刷新 `tasks:list`，或直接 `invalidateQueries("tasks:list")`。
- 合并状态优先级：当 `lib.isFullySynced === true` 或 `lib.completionPercent >= 100` 时，优先展示完成态，不能再让旧的 `activeTask` 覆盖按钮 / badge / 进度条。
- 对任务完成或失败状态增加显式清理规则，避免 stale task 长时间占用 UI。

落地顺序：
- 先修 `tasks:list` 的刷新链路。
- 再统一 `buttonLabel / badgeVariant / progressText` 的状态计算函数。
- 最后回归“同步中 -> 100% -> 无需扫描”的完整切换链路。

状态：
- `已实现，待人工复核`

### F-032 通用排序条应去掉刷新按钮

现象：
- 当前库页、喜欢页、演员关联影片页的通用排序条仍保留“刷新”按钮。
- 当前反馈希望选择排序项后直接生效，不再保留单独刷新动作。

已核对证据：
- `tauri/src/components/shared/QueryToolbar.tsx` 当前无条件渲染刷新按钮。
- `LibraryPage.tsx`、`FavoritesPage.tsx`、`ActorDetailPage.tsx` 都给 `QueryToolbar` 传了 `onRefresh`，但 `handleSortChange` 本身已经在更新 router query 并触发重新取数。
- `QueryToolbar` 内部 `handleSortSelect` 当前已经直接调用 `onSortChange`，说明排序并不依赖额外提交。

根因判断：
- 当前工具条仍带着“筛选后手动刷新”的旧交互模型，但结果页现在其实已经是响应式查询模型。
- 刷新按钮因此从“必要动作”退化成了“重复动作”。

建议解决方案：
- 将刷新按钮从共享 `QueryToolbar` 中移除。
- `onRefresh` 从共享接口中删除，或保留为非共享场景的可选能力，但默认不渲染。
- 如果个别页面确实需要“重新拉取”，应由页面自身以单独动作条提供，而不是放进通用排序工具条。

落地顺序：
- 先修改 `QueryToolbar` 组件 API 和 UI。
- 再清理库页、喜欢页、演员详情页的 `handleRefresh` 接线。
- 最后回归搜索、排序、分页仍能自动刷新数据。

状态：
- `已实现，待人工复核`

### F-033 通用分页组件应固定在活动页右下角

现象：
- 当前分页组件会随着内容多少和内容流高度变化而上下漂移。
- 当前反馈希望分页组件固定在活动页窗口右下角。

已核对证据：
- `LibraryPage.tsx`、`FavoritesPage.tsx`、`ActorDetailPage.tsx` 当前都把 `<Pagination />` 直接渲染在内容网格之后。
- `tauri/src/pages/pages.css` 中 `.pagination-bar` 只有 `justify-content: flex-end; padding: 16px 0;`，并没有固定底部定位能力。
- `tauri/src/App.css` 的 `.content-area` 当前是单层滚动区，没有分页专属 footer slot。

根因判断：
- 根因不是分页组件逻辑，而是分页组件的挂载层级仍处于内容流内部。
- 只要分页留在列表 DOM 后面，它就必然会跟着内容高度移动。

建议解决方案：
- 在活动页共享 shell 中增加固定 footer 区。
- 内容网格放到 `page-scroll-body`，分页组件放到 `page-footer-right`，并使用 `position: sticky; bottom: 0;` 或 flex footer 方式固定在右下角。
- 底部分页区预留稳定高度，即使结果较少时也保持同样的右下角锚点。

落地顺序：
- 先配合 `F-028` 引入活动页 shell / body / footer 分层。
- 再迁移库页、喜欢页、演员详情页的分页挂载位置。
- 最后回归单页、多页、空页和窄窗宽度表现。

状态：
- `已实现，待人工复核`

### F-036 影片卡应支持小 / 中 / 大三档显示大小

现象：
- 当前影片卡大小固定，无法根据窗口尺寸和个人偏好调整显示密度。
- 当前需求是提供小 / 中 / 大三档：
- 小：当前大小。
- 中：当前海报区域的 2 倍。
- 大：当前海报区域的 3 倍。
- 库页、喜欢页、演员关联影片页统一生效。
- 设置页显示页签需要提供三档示意框。
- 卡片文字区高度保持不变，只放大海报显示区。

已核对证据：
- `tauri/src/components/shared/VideoCard.css` 当前卡片尺寸完全由固定样式控制，文本区也跟卡片整体耦合。
- `tauri/src/pages/pages.css` 中 `.video-grid` 当前使用 `repeat(auto-fill, minmax(180px, 1fr))`，只有单一列宽策略。
- `tauri/src/pages/SettingsPage.tsx` 当前 `SETTING_GROUPS` 中没有“显示”分组。
- `tauri/src/api/types.ts` 的设置 DTO 当前也没有任何显示密度或影片卡尺寸字段。

根因判断：
- 这不是单页样式修补，而是共享卡片尺寸体系、网格排布、设置持久化和设置页预览的联动需求。
- 当前实现里“海报区大小”和“卡片整体布局”没有被拆开，因此无法只放大海报区、保持文字区高度不变。

建议解决方案：
- 在设置体系中新增 `display` 分组和 `videoCardSize` 字段，值域为 `small | medium | large`。
- 在 `VideoCard` 共享组件中引入尺寸变体，只放大海报区宽高，文本区使用固定高度 token。
- 在库页、喜欢页、演员详情页的网格容器上统一使用尺寸变量驱动列宽，并保留自动换行 / 自动补位能力。
- 在设置页新增显示页签，提供三档示意框和即时预览样式说明。

落地顺序：
- 先扩展前后端设置 DTO 与默认值。
- 再改 `VideoCard` / `video-grid` 的尺寸变量体系。
- 最后接入设置页 UI 和三档预览框。

状态：

### F-037 删除影片应使用通用提示框，且必须真正删除本地原片与搜刮数据

现象：
- 当前删除影片时仍弹出浏览器原生确认框。
- 确认文案不符合预期，目标文案应为“删除影片原片和搜刮数据”，且支持多语言。
- 实际删除后本地目录没有被真正删除。

已核对证据：
- `tauri/src/pages/LibraryPage.tsx`、`FavoritesPage.tsx`、`ActorDetailPage.tsx` 当前都直接调用原生 `confirm(tc("deleteVideoConfirm"))`。
- 项目里已经存在共享确认框组件 `tauri/src/components/shared/ConfirmDialog.tsx`。
- `tauri/src/api/client.ts` 的 `deleteVideo(videoId, deleteFile?)` 和 `batchDelete(..., deleteFiles?)` 默认只有传 `true` 才会拼接删除文件参数。
- 前端当前删除调用没有传 `true`，因此 Worker 侧 `VideosController.DeleteVideo` 接收到的 `deleteFile` 默认为 `false`。
- `dotnet/Jvedio.Worker/Services/VideoService.cs` 中 `DeleteVideo(videoId, deleteFile)` 只有在 `deleteFile == true` 时才会删除视频文件和 sidecar；即便删文件成功，当前也不会继续清理空目录。

根因判断：
- 这是“确认交互不统一 + 删除 API 默认语义过软 + Worker 未清理目录”叠加出来的问题。
- 当前前端删的是“数据库记录”，用户理解的却是“删除原片和搜刮数据”，两者语义完全不一致。

建议解决方案：
- 前端所有删除影片入口统一改用 `ConfirmDialog`。
- 新增中英文多语言键，明确文案为“删除影片原片和搜刮数据”以及对应批量版本。
- 当用户确认删除时，前端统一调用 `deleteVideo(videoId, true)` / `batchDelete(..., true)`。
- Worker 删除链路在删除视频文件和 sidecar 后，继续尝试删除已空的影片目录；若未来需要“只从库中移除”，必须单独设计为另一条明确动作。

落地顺序：
- 先替换前端确认框和多语言文案。
- 再修正前端删除接口调用参数。
- 最后补 Worker 目录清理和结果回执字段，并做单删 / 批删回归。

状态：
- `已实现，待人工复核`
